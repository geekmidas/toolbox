# Testing SST (ion) constructs

> Status: **design / deferred**. This captures the approach and the research
> behind it; the tooling (`@geekmidas/testkit/pulumi` + `@geekmidas/testkit/sst`)
> is not built yet. Companion to [`sst-constructs.md`](./sst-constructs.md).

## 1. Goal

Test the `@geekmidas/cloud/sst` constructs by asserting on the **real Pulumi
resource graph** they register — the actual resources (Lambdas, integrations,
permissions, routes), their types, and their resolved inputs — **without
deploying**. Explicitly **not** shallow stubs that only record the arguments
handed to a faked base class: those don't validate what SST/Pulumi actually
produces (provider input shapes, transforms, derived resources).

## 2. Background: why this is hard

SST **v1/v2** ran on AWS CDK, so you could synthesize CloudFormation and assert
on it (`template.hasResourceProperties('AWS::Lambda::Function', { … })`).

SST **v3/v4 (ion)** runs on **Pulumi**, which has no synth-to-template step and
a completely different testing model. SST ion ships **no official test harness**
— see the open issue [sst#6370 "How to Unit Test Infrastructure
Definitions"](https://github.com/anomalyco/sst/issues/6370) (no maintainer
answer as of writing). So we build our own.

## 3. The mechanism: Pulumi `runtime.setMocks`

Pulumi's [unit-testing](https://www.pulumi.com/docs/iac/guides/testing/unit/)
runs the **real** Pulumi engine in-memory and only mocks the cloud-provider CRUD:

```ts
import * as pulumi from '@pulumi/pulumi';

pulumi.runtime.setMocks({
  newResource: (args) => ({ id: `${args.name}-id`, state: args.inputs }),
  call: (args) => args.inputs,
});
// import the program/construct AFTER setMocks, then assert on Outputs via .apply()
```

`newResource` is invoked for **every** resource the construct registers, with
its **real** inputs — that is the "correct Pulumi state" we want to capture and
assert on. Caveat from Pulumi's docs: mocks do **not** run stack transforms or
lifecycle hooks on their own; SST registers transforms (tags, retain-on-delete)
in its `run()` wrapper, so for full fidelity the harness should drive SST's own
`run()` pipeline (§5), not just call `setMocks` and `new`.

## 4. Package layout

Two layers under the existing `@geekmidas/testkit` (which already exposes many
subpaths: `./aws`, `./kysely`, `./postgres`, …):

- **`@geekmidas/testkit/pulumi`** — generic, provider-agnostic foundation. Wraps
  `runtime.setMocks`, runs a program/thunk, and collects the registered resource
  graph into an assertable structure (`{ type, name, urn, inputs }[]` with
  helpers like `findResources(type)`). Robust and documented; reusable for any
  Pulumi program.
- **`@geekmidas/testkit/sst`** — builds on `/pulumi` and bootstraps SST's ambient
  globals so real `sst.aws.*` components run (§5). Provides a harness like
  `await withSstApp({ name, stage }, () => new Api(stack, 'Api', props))` →
  returns the captured resources.

## 5. Bootstrapping the SST globals

SST components depend on globals the CLI injects; nothing instantiates them
standalone without this. Two sources, both visible in a consuming app's
`.sst/platform`:

- **Shim exports** — `.sst/platform/src/shim/run.js` re-exports these as the
  `$`-globals (esbuild injects them): `$util` (= `@pulumi/pulumi`), `$output`,
  `$apply`, `$resolve`, `$interpolate`, `$concat`, `$jsonParse`,
  `$jsonStringify`, `$linkable`, `$asset`, `$config`, `$transform`, `$secrets`.
- **Runtime globals** — declared in `.sst/platform/src/global.d.ts`: `$app`
  (`Simplify<$APP>` — `name`, `stage`, `removal`, `protect`, …), `$dev`
  (`boolean`), `$cli` (`{ paths, … }`). The CLI sets these on the global scope
  before running `sst.config.ts`; the harness must set equivalents
  (`globalThis.$app = { name, stage, … }`, etc.).

For transform fidelity, the harness should also call SST's
`.sst/platform/src/auto/run.ts` `run()` (it does `Function.reset()` /
`Link.reset()` and registers the stack transformations) rather than invoking the
program directly.

## 6. The make-or-break constraint: one `@pulumi/pulumi` instance

`runtime.setMocks` only intercepts resources registered through the **same
`@pulumi/pulumi` module instance** it was called on.

- SST's components `import "@pulumi/pulumi"`, which resolves to
  **`.sst/platform/node_modules/@pulumi/pulumi`** — a per-app, version-pinned
  copy generated by `sst install`.
- There is **no `@pulumi/pulumi` in the normal workspace** at all (only inside
  `.sst/platform`).

Therefore the harness **must** call `setMocks` on *that exact* copy, or the
mocks silently fail to intercept and capture nothing. Concretely, the consuming
app's vitest config must `resolve.alias` `@pulumi/pulumi` (and likely `@pulumi/*`
providers) to the `.sst/platform/node_modules` copy, and `testkit/pulumi` must
import pulumi through the same alias. **This singleton sharing is the first thing
to validate** (§8) — it determines whether real-state capture is achievable at
all.

## 7. Test tiers (what to test where)

1. **Pure-logic unit tests — available now, plain vitest, no Pulumi/SST.**
   `App`, `Stack`, `naming` are runtime-pure (they reference SST only via erased
   type annotations like `$util.Output`). The env validation is already pure and
   tested in `@geekmidas/envkit`. **Action:** extract `Api`'s decisions
   (route-key building, the `_id`/`_type`→`{type}` link bridge,
   `route.runtime ?? apiRuntime ?? 'nodejs24.x'`, least-privilege link selection,
   per-route validation/aggregation) into standalone helpers and unit-test them.
   This is the highest-ROI layer and matches Pulumi's own guidance ("structure
   complex logic as separate testable functions").
2. **Real-state Pulumi tests — this document.** `testkit/pulumi` + `testkit/sst`
   capture the registered resource graph for assertions like "the API registered
   N integration resources, each Lambda has `runtime: nodejs24.x` and these env
   vars / links".
3. **Integration tests — few, critical paths only.** `sst deploy --stage pr-xxx`
   to an ephemeral stage (real AWS, or LocalStack where supported) + assert via
   the AWS SDK or Pulumi Automation API. Slow, needs credentials.

Plus the existing cheap layer: the **`ts:check:sst`** gate already catches wrong
SST API usage / arg shapes / runtime strings against the real SST v4 types.

Note: `packages/cloud` currently has **no `vitest.config`**, so its tests aren't
discovered under the root `projects: ['packages/*']` — add one when tiers 1–2
land.

## 8. Validation spike (do this before building the packages)

From `packages/cloud` (it already has `.sst/platform`):

1. Import `@pulumi/pulumi` from `.sst/platform/node_modules`, call `setMocks`,
   register a trivial `aws.s3.Bucket`, and confirm `newResource` fires and the
   resource is captured. (Proves the engine + capture work.)
2. Bootstrap the §5 globals, import a real `sst.aws.Function`, instantiate it,
   and confirm its underlying AWS resources are captured via `newResource`.
   (Proves the SST bootstrap + singleton sharing.)

If both pass, build `testkit/pulumi` then `testkit/sst`. If step 2 fights us
(singleton/global coupling), fall back to tier-1 + tier-3 and revisit.

## 9. Risks & open questions

- **Coupling to `.sst/platform` internals** (shim export names, `$app`/`$cli`
  shapes, `auto/run.ts`) — these are generated and can change across SST
  versions. The harness is pinned to whatever `sst install` produced.
- **Per-app pulumi copy** — the alias path is app-specific; `testkit/sst` needs a
  documented vitest setup the consuming app applies.
- **Transform fidelity** — decide whether to drive SST's `run()` (full
  transforms) or accept `setMocks`-only (no stack transforms).
- **`@pulumi/pulumi` dependency** — `testkit/pulumi` needs it as a peer/dev dep
  for its own typing/standalone use, but `testkit/sst` must defer to the app's
  `.sst/platform` copy at runtime.
